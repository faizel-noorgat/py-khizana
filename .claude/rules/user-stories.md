# User Stories Completion Rules

This file defines how to complete each section in `user-stories.yaml` during Phase 01 (PRD Discovery).

## user_stories

**When:** Section 3-4 - Generate Personas and User Stories (output from `/generate-user-stories` skill)

**How to complete:**
- Each story is written from a persona's first-person perspective
- Generated by persona agents spawned by the `/generate-user-stories` skill
- ID format: `US-001`, `US-002`, etc.
- Links to persona via `persona_id` (P-XXX from product-context.yaml)

**Story format:**
```yaml
user_stories:
  - id: "US-001"
    persona_id: "P-001"
    title: "Quick document search"
    story: "As a Digital Hoarder, I want to find any document in seconds so I don't waste time navigating folders."
    acceptance_criteria:
      - "Search returns results within 2 seconds"
      - "Search works across all document types"
      - "Results show relevant matches first"
    priority: "P0"
    status: "PENDING"
    notes: ""
```

**Fields:**

| Field | Description |
|-------|-------------|
| `id` | US-XXX format |
| `persona_id` | Links to P-XXX in product-context.yaml |
| `title` | Short name for the story |
| `story` | "As a [persona], I want [goal] so that [benefit]" |
| `acceptance_criteria` | Testable conditions for "done" |
| `priority` | P0 (must-have), P1 (important), P2 (nice-to-have) |
| `status` | PENDING | IN_PROGRESS | COMPLETE | REMOVED |
| `notes` | Parking lot for overflow (process at Section 12) |

**Relationship to features:**
- User stories are discovery tools that spark feature ideas
- Features in `product-context.yaml` are derived from user stories
- A single feature may address multiple user stories
- Use `requested_by` in features to link back to personas

**Notes parking lot:**
- During discovery, overflow info may be parked in `notes`
- At Section 12, process `notes` and move to correct fields or files
- `notes` should be empty before handoff

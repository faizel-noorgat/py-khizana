# Product Context Completion Rules

This file defines how to complete each section in `product-context.yaml` during Phase 01 (PRD Discovery).

## Section Completion Order

Sections are filled incrementally during discovery conversation. Order follows natural conversation flow, not file order.

---

## stakeholders

**When:** Section 5 - Map Stakeholders and Users

**How to complete:**
- Identify everyone who influences requirements, approves work, or is affected by outcome
- Not just end users — include decision-makers, approvers, impacted teams
- Ask: "Who else has influence or is affected besides the primary users?"
- Roles: Decision-maker, Approver, Impacted team, Support/maintenance, External partner

**Example:**
```yaml
stakeholders:
  - role: "Decision-maker"
    person_or_team: "Solo founder"
  - role: "End user"
    person_or_team: "Book enthusiasts with scattered collections"
```

---

## target_users

**When:** Section 5 - Map Stakeholders and Users

**How to complete:**
- Link to personas generated by `/generate-personas` skill
- Each user links to a persona (P-XXX) from product-context
- Fill `goals`, `pain_points`, `technical_level` based on persona details
- ID format: `USR-001`, `USR-002`, etc.
- `technical_level`: LOW | MEDIUM | HIGH

**Example:**
```yaml
target_users:
  - id: "USR-001"
    persona: "P-001"
    description: "Digital Hoarder — collects digital content across platforms"
    goals:
      - "Find any book instantly"
      - "See what I've already read"
    pain_points:
      - "Forgot which books I own"
      - "Duplicates across platforms"
    technical_level: "MEDIUM"
```

---

## user_needs

**When:** Section 5 - Map Stakeholders and Users

**How to complete:**
- What users need to accomplish (not what system does)
- Derived from persona goals and pain points
- Focus on outcomes, not features
- ID format: `UN-001`, `UN-002`, etc.

**Example:**
```yaml
user_needs:
  - id: "UN-001"
    description: "Find any book in my collection quickly"
  - id: "UN-002"
    description: "Track reading progress without friction"
  - id: "UN-003"
    description: "Discover what to read next based on what I already like"
```

---

## features

**When:** Section 6 - Capture Requirements

**How to complete:**
- System capabilities derived from user needs and user stories
- Each feature must link to a need or story
- Assign priority: P0 (must-have), P1 (important), P2 (nice-to-have)
- Ask: "What would you cut if we ran out of time?" to surface true priorities
- ID format: `FTR-001`, `FTR-002`, etc.
- Status: PENDING | IN_PROGRESS | COMPLETE | REMOVED

**Example:**
```yaml
features:
  - id: "FTR-001"
    description: "Add/edit/delete books in library"
    priority: "P0"
    status: "PENDING"
  - id: "FTR-002"
    description: "Search books by title, author, or tags"
    priority: "P0"
    status: "PENDING"
  - id: "FTR-003"
    description: "Track reading progress and history"
    priority: "P1"
    status: "PENDING"
  - id: "FTR-004"
    description: "Generate reading suggestions"
    priority: "P1"
    status: "PENDING"
```

---

## user_flows

**When:** Section 6 - Capture Requirements (optional, for complex flows)

**How to complete:**
- Document multi-step journeys through the system
- Focus on P0 features and core workflows
- Number steps sequentially
- ID format: `FLW-001`, `FLW-002`, etc.

**Example:**
```yaml
user_flows:
  - id: "FLW-001"
    name: "Add a new book"
    steps:
      - step: 1
        description: "User searches for book by title/author"
      - step: 2
        description: "System fetches metadata from API"
      - step: 3
        description: "User confirms and adds to library"
```

---

## requirements.functional

**When:** Section 6 - Capture Requirements (if formal specs needed)

**How to complete:**
- Detailed functional requirements when features need decomposition
- Use when P0 feature is complex or ambiguous
- ID format: `FR-001`, `FR-002`, etc.
- Status: PENDING | IN_PROGRESS | COMPLETE

**Example:**
```yaml
requirements:
  functional:
    - id: "FR-001"
      description: "System must persist book metadata locally"
      status: "PENDING"
    - id: "FR-002"
      description: "System must sync changes to local storage"
      status: "PENDING"
```

---

## requirements.non_functional

**When:** Section 6 - Capture Requirements

**How to complete:**
- Performance, security, accessibility, compliance requirements
- Each must have a **measurable_target** (not vague terms)
- Category: PERFORMANCE | SECURITY | ACCESSIBILITY | COMPLIANCE
- Ask: "What non-functional needs could cause launch failure?"
- ID format: `NFR-001`, `NFR-002`, etc.
- Verification status: PENDING | PASS | FAIL

**Example:**
```yaml
requirements:
  non_functional:
    - id: "NFR-001"
      category: "PERFORMANCE"
      description: "Book search response time"
      measurable_target: "Under 500ms for local search"
      verification_status: "PENDING"
    - id: "NFR-002"
      category: "SECURITY"
      description: "Local data storage"
      measurable_target: "All data stored locally, no cloud sync without consent"
      verification_status: "PENDING"
```

## risks

**When:** Section 10 - Surface Risks, Assumptions, and Open Questions

**How to complete:**
- What could go wrong and its potential impact
- Assign likelihood and impact: LOW | MEDIUM | HIGH
- Document mitigation strategy if known
- Ask: "What's the one thing that would derail this project if it went wrong?"
- ID format: `R-001`, `R-002`, etc.

**Example:**
```yaml
risks:
  - id: "R-001"
    description: "Google Books API rate limits could block metadata fetches"
    likelihood: "MEDIUM"
    impact: "HIGH"
    mitigation: "Implement local caching, fallback to manual entry"
  - id: "R-002"
    description: "User loses local data without backup"
    likelihood: "LOW"
    impact: "HIGH"
    mitigation: "Add export feature in Phase 2"
```

---

## assumptions

**When:** Section 10 - Surface Risks, Assumptions, and Open Questions

**How to complete:**
- What we're taking as given without verification
- Mark `validated: true` only after confirmed with user or research
- Ask: "What are we assuming that might not be true?"
- ID format: `A-001`, `A-002`, etc.

**Example:**
```yaml
assumptions:
  - id: "A-001"
    description: "Users have internet access for initial book search"
    validated: false
  - id: "A-002"
    description: "Users primarily track physical books, not ebooks"
    validated: true
```

---

## open_questions

**When:** Section 10 - Surface Risks, Assumptions, and Open Questions

**How to complete:**
- Unknowns that would change scope or approach
- Triage as blocking (must resolve before Phase 02) or non-blocking
- Status: OPEN | ANSWERED | DEFERRED
- Capture answer when resolved
- ID format: `OQ-001`, `OQ-002`, etc.

**Example:**
```yaml
open_questions:
  - id: "OQ-001"
    description: "Should suggestions include books not yet in library?"
    status: "OPEN"
    blocking: false
    answer: ""
  - id: "OQ-002"
    description: "What import formats should we support?"
    status: "OPEN"
    blocking: true
    answer: ""
```

---

## glossary

**When:** Section 11 - Establish Key Terms

**How to complete:**
- Domain-specific terms user introduced with specific meaning in this context
- Capture terms that could be interpreted multiple ways
- Prevents misunderstanding across later phases
- Ask: "Are there terms you're using that might mean something different to others?"

**Example:**
```yaml
glossary:
  - term: "Library"
    definition: "User's personal collection of tracked books, not a public lending library"
  - term: "Reading status"
    definition: "Current state: Want to read, Reading, Finished, Abandoned"
```

---

## notes

**When:** Any time during discovery

**How to complete:**
- Free-form capture space for anything not fitting structured fields
- Use for partial thoughts, user anecdotes, conversation notes
- Does not need structure — raw capture

## Anti-Pattern Checklist

Before finalizing any section, verify:

| ❌ Avoid | ✓ Use instead |
|----------|---------------|
| "fast" | "response time under 200ms" |
| "easy to use" | "user completes task in under 3 clicks" |
| "user-friendly" | "user satisfaction score above 4.0" |
| "intuitive" | "new user completes core flow without help" |
| "scalable" | "handles 10,000 concurrent users" |
| "secure" | "all data encrypted at rest with AES-256" |
| "reliable" | "99.9% uptime" |

**Validation questions:**
- Can a developer implement this without clarification?
- Can a tester write a pass/fail test?
- If you replaced the vague term with a number, would implementation change?

If any field fails, rephrase before documenting.
